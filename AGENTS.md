## Code Canonicality

Same logic must have exactly one implementation in the repo. Version evolution belongs in git history, not in filenames or identifiers.

- Before adding any function, class, module, or file: grep the codebase for existing similar implementations. If found, extend or refactor the existing one. Only create new if truly absent.
- Forbidden suffixes for filenames, class names, function names, and module names: `_v1`, `_v2`, `_v3`, `_new`, `_old`, `_backup`, `_temp`, `_copy`, `_final`, `_real`, `_improved`, `_refactored`, `_fixed`, `_legacy`, `_deprecated`. If one exists in the codebase already, treat it as debt to unify, not a pattern to extend.
- Refactor in place: commit the working state first, modify the original file, run tests. If it breaks, `git revert`. Never keep an old version alongside the new "just in case" — git is the safety net.
- Commented-out code is a lint error, not a safety net. Delete it; git remembers.
- Unreferenced exports and dead code paths are debt. Wire them up in the same change or delete them.
- Do not use the repo as a scratchpad. Directories like `tmp/`, `scratch/`, `backup/`, `_old/`, `deprecated/`, `archive/`, `wip/` must not appear in commits.

### Dependency policy

- **Lockfile**: `sdk/package-lock.json` committed. Agents must run `cd sdk && npm ci` after pulling and never bypass it.
- **Updates**: Renovate configured (`renovate.json`) with 3-day minimum release age. Major versions require approval.
- **Forbidden**: no `npm install <pkg>@latest` in PRs without justification.
- **New dependency**: must justify in PR description — what it does and why existing tools can't do it.

### TypeScript-specific rules

- **`tsconfig.json`**: `"strict": true`, `"noUncheckedIndexedAccess": true`, `"exactOptionalPropertyTypes": true`, `"noImplicitOverride": true`. No `// @ts-ignore` without a paired `// TODO(<ticket>):` comment.
- **ESM**: local TypeScript imports use `.js` extensions. Use `import type` for type-only imports.
- **Linter**: ESLint with `@typescript-eslint`, `eslint-plugin-import`. Strict mode: warnings are errors in CI.
- **Formatter**: Prettier. `lint-staged` runs Prettier + ESLint on changed files in pre-commit.
- **Anti-drift**: duplicate code via `jscpd` (threshold 2%), dead exports via `knip`, conventional commits via `commitlint`.
- **Size discipline**: source files over **700 effective lines** and cyclomatic complexity over **10** fail ESLint. Split by feature, domain boundary, or adapter responsibility before raising limits.
- **Tests**: Node built-in `node:test`. Test files `*.test.js` in `sdk/test/`. Coverage gate: **90%**.
- **Imports**: no barrel files re-exporting everything — they break tree-shaking. Direct imports only.
- **Types over interfaces** for unions/intersections; **interfaces over types** for extensible object shapes.
- **Forbidden**: `any` (use `unknown`); `as` casts without justification; `@ts-ignore`; `@ts-expect-error` without explanation.

### Common pitfalls (lint-enforced)

- Implicit `any` → tsconfig `noImplicitAny`.
- Untyped `catch` → tsconfig `useUnknownInCatchVariables`.
- Circular imports → `import/no-cycle`.
- Unused imports → ESLint.
- File line count > 700 → `eslint` `max-lines`.
- Cyclomatic complexity > 10 → `eslint` `complexity`.
- `console.log` in production code → ESLint `no-console`.

### Universal stack rules

- Lockfile is the single source of truth for resolved versions. Do not bypass it.
- New dependencies require a justification line in the PR description.
- Major-version upgrades require human review even if dependency-update automation is configured.
- A package introduced but unused in the same PR is a CI failure (knip).
- License audit: dependencies must use MIT, Apache-2.0, BSD-3-Clause, or ISC.

## Conventions

### TypeScript

- Package is ESM-first. Local TypeScript imports use `.js` extensions.
- Use `import type` for type-only imports.
- Keep public contracts in `sdk/src/domain/types.ts` and errors in `sdk/src/domain/errors.ts`.
- `KnowledgeBase` orchestration belongs in `sdk/src/application/knowledge-base.ts`; avoid moving vendor/provider logic there.
- Infrastructure modules implement ports: `SourceParser`, `SearchAdapter`, `LLMProvider`.
- File writes should use `atomicWrite()` from `sdk/src/infrastructure/filesystem.ts` for SDK-managed bundle files.
- Provider SDK imports (`openai`, `@anthropic-ai/sdk`) must stay in `sdk/src/infrastructure/providers/` only. Application/domain code depends on `LLMProvider`, not vendor clients.
- Mutating SDK APIs return `ChangeSet` with `created`, `updated`, `deleted`, `skipped`, `failed`, and `warnings`; do not replace this with thrown errors for expected per-file failures.
- `query()` must not fabricate citations. Keep citation filtering bundle-local and restricted to retrieved paths.
- Preserve unknown OKF frontmatter fields and unknown `type` values; do not model OKF `type` as a closed enum.
- Deterministic methods (`search`, `validate`, `status`, `listConcepts`, `export`) must work without an LLM provider.
- LLM-dependent methods must fail with `ConfigurationError` when no provider is configured.

### Testing

- Framework: Node built-in `node:test` plus `node:assert/strict`.
- Tests live in `sdk/test/` and exercise public APIs via built `dist` output.
- Provider tests use fake official-client objects; they do not call live OpenAI/Anthropic APIs.
- SDK tests import from `../dist/index.js` — keep build output current when testing locally.
- Add or update tests for behavior changes: validation rules, frontmatter parsing, citation filtering, adapter boundaries.
- Do not use mocks for adapter integration tests where local filesystem/parser/search behavior is the point under test.
- TDD required: every new production file must have a paired test file.

### Key files

- `sdk/src/index.ts` — public SDK exports.
- `sdk/src/application/knowledge-base.ts` — main SDK facade.
- `sdk/src/domain/types.ts` — public SDK types/interfaces.
- `sdk/src/domain/errors.ts` — SDK error classes.
- `sdk/src/infrastructure/providers/openai.ts` and `anthropic.ts` — provider adapters.
- `sdk/test/knowledge-base.test.js` — SDK regression tests.

## Important Development Notes

AI-generated code commonly gets these wrong in this codebase:

1. **ESM import extensions**: Always use `.js` extensions in local TypeScript imports (e.g., `import { foo } from "./bar.js"`), even though the source file is `.ts`. This matches Node.js ESM resolution.
2. **Layer discipline**: Application code (`sdk/src/application/`) must NOT import from infrastructure (`sdk/src/infrastructure/`). Depend on domain ports (`LLMProvider`, `SearchAdapter`, `SourceParser`), not concrete implementations (`openai`, filesystem utilities).
3. **OKF type openness**: Do NOT model OKF `type` as a closed enum. Preserve unknown frontmatter fields and unknown `type` values. The SDK must pass through unrecognized metadata.
4. **Citation integrity**: `query()` must NOT fabricate citations. Filter citations to bundle-local paths actually in the retrieved set.
5. **Deterministic vs LLM methods**: `search()`, `validate()`, `status()`, `listConcepts()`, `export()` must work without an LLM provider. `query()` and synthesis methods must fail with `ConfigurationError` when no provider is configured.
6. **ChangeSet over exceptions**: Mutating APIs return `ChangeSet` with per-file success/failure collections. Do NOT use thrown errors for expected per-file failures (e.g., parse errors on one of many documents).

## Agent Operating Rules

### Permissions

- Agents may commit directly to feature branches.
- Agents may never push to `main` — all changes go through PR.
- Agents may never force-push (`git push --force` or `-f`).
- Agents may never use `--no-verify` to bypass hooks.
- Agents may never deploy to production or modify deployment configs.

### Commit discipline

- All commits must follow conventional commit format: `type(scope): description`.
- Allowed types: `feat`, `fix`, `refactor`, `chore`, `docs`, `test`, `ci`, `build`, `perf`.
- Agent commits must include `Co-authored-by: Agent <agent@llm-wiki>` in the commit body.
- Conventional commits enforced by `commitlint` in pre-commit hook.

### Evidence requirements

Every PR must include runtime evidence in the `## Runtime evidence` section:

- Type system: `just typecheck` output showing exit 0.
- Tests: `just test` output showing pass count.
- Lint: `just lint` output showing 0 warnings.
- For SDK behavior changes: real consumer workflow evidence (create → ingest → search → validate).
- Live OpenAI/Anthropic API calls are not required in PR evidence; provider-boundary checks are separate E2E checks.

### Secrets handling

- `.env` files are never committed; Claude Code settings block reading them.
- API keys belong in environment variables, never in source.
- `constraints.yaml` exemptions allow `.env.example` for documentation.

### Code review

- Self-review your own diff before marking a PR ready.
- Critical path changes require explicit human approval.
- No PR may merge without passing CI (all layers).
