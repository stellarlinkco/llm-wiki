## [TLDR](https://stellarlink.co/articles/2025-%E6%88%91%E8%AE%A9-agent-%E8%87%AA%E5%B7%B1%E8%B7%91%E5%AE%8C%E4%BA%86#tldr)

从 6 月用 Claude Code 手动盯着干活,到 11 月云端触发,再到 12 月全自动开发流程,花了半年时间让 AI 从”需要人盯着”变成”自己能跑完”。

现在能做到:丢个需求 → 确认 PRD → 自动拆分并发执行 → 自动 review 修复 → 合并上线 → 测试闭环。

整个过程只需要在 PRD 阶段确认一次。

## [从手动等待到全自动](https://stellarlink.co/articles/2025-%E6%88%91%E8%AE%A9-agent-%E8%87%AA%E5%B7%B1%E8%B7%91%E5%AE%8C%E4%BA%86#%E4%BB%8E%E6%89%8B%E5%8A%A8%E7%AD%89%E5%BE%85%E5%88%B0%E5%85%A8%E8%87%AA%E5%8A%A8)

6 月开始用 Claude Code,体验到了比 Cursor 更直接的 vibe coding。丢个需求给它,自己去read、write,挺爽的。但有几个问题让人难受:

1.  **权限卡点** - 遇到权限问题需要人工按回车确认
2.  **被动等待** - 代码写完了需要人工去 review
3.  **串行阻塞** - 必须盯着它干完活才能操作下一步

每次坐在电脑前等 Claude Code 工作结束,就会想:能不能把我分身出来盯着它,我自己去干别的?

11 月 2 日发布了 swe-agent.ai,把这个想法落地了:在 GitHub issue 里 comment `/code` 就能触发后台的 Claude Code。人不用守在电脑前了,但还是需要每次改动都去 comment 触发一下。从”守在电脑前”变成了”守在 GitHub 里”,换了个空间继续被限制。

## [三次迭代:从单次任务到全流程自动化](https://stellarlink.co/articles/2025-%E6%88%91%E8%AE%A9-agent-%E8%87%AA%E5%B7%B1%E8%B7%91%E5%AE%8C%E4%BA%86#%E4%B8%89%E6%AC%A1%E8%BF%AD%E4%BB%A3%E4%BB%8E%E5%8D%95%E6%AC%A1%E4%BB%BB%E5%8A%A1%E5%88%B0%E5%85%A8%E6%B5%81%E7%A8%8B%E8%87%AA%E5%8A%A8%E5%8C%96)

### [Claude Code:手动盯着干活](https://stellarlink.co/articles/2025-%E6%88%91%E8%AE%A9-agent-%E8%87%AA%E5%B7%B1%E8%B7%91%E5%AE%8C%E4%BA%86#claude-code%E6%89%8B%E5%8A%A8%E7%9B%AF%E7%9D%80%E5%B9%B2%E6%B4%BB)

用 Claude Code 时的典型流程:

```
需求 → 等待分析 → 等待写代码 → 人工 review → 人工提交
```

优点是直接,缺点是离不开人。

### [swe-agent 1.0:云端触发](https://stellarlink.co/articles/2025-%E6%88%91%E8%AE%A9-agent-%E8%87%AA%E5%B7%B1%E8%B7%91%E5%AE%8C%E4%BA%86#swe-agent-10%E4%BA%91%E7%AB%AF%E8%A7%A6%E5%8F%91)

把 Claude Code 搬到云端,通过 GitHub webhook 触发:

```
Issue comment /code → 后台 Claude Code 执行 → 代码提交 → 结束
```

解决了”守在电脑前”的问题,但每次还是需要人工去触发下一步。

### [swe-agent 2.0:全流程自动化](https://stellarlink.co/articles/2025-%E6%88%91%E8%AE%A9-agent-%E8%87%AA%E5%B7%B1%E8%B7%91%E5%AE%8C%E4%BA%86#swe-agent-20%E5%85%A8%E6%B5%81%E7%A8%8B%E8%87%AA%E5%8A%A8%E5%8C%96)

核心突破:从”单次任务”升级为”流程编排”。

```
需求 → PRD (人工确认) → 任务拆分 → 并发执行 → Review → 修复 → 合并 → 测试 → 完成
```

整个流程只在 PRD 阶段需要人确认一次,后面全自动。

## [两种模式,各干各的事](https://stellarlink.co/articles/2025-%E6%88%91%E8%AE%A9-agent-%E8%87%AA%E5%B7%B1%E8%B7%91%E5%AE%8C%E4%BA%86#%E4%B8%A4%E7%A7%8D%E6%A8%A1%E5%BC%8F%E5%90%84%E5%B9%B2%E5%90%84%E7%9A%84%E4%BA%8B)

### [单次任务模式](https://stellarlink.co/articles/2025-%E6%88%91%E8%AE%A9-agent-%E8%87%AA%E5%B7%B1%E8%B7%91%E5%AE%8C%E4%BA%86#%E5%8D%95%E6%AC%A1%E4%BB%BB%E5%8A%A1%E6%A8%A1%E5%BC%8F)

适合快速修 bug 或小改动,评论一下就能触发,改完就结束。

### [全流程模式](https://stellarlink.co/articles/2025-%E6%88%91%E8%AE%A9-agent-%E8%87%AA%E5%B7%B1%E8%B7%91%E5%AE%8C%E4%BA%86#%E5%85%A8%E6%B5%81%E7%A8%8B%E6%A8%A1%E5%BC%8F)

适合完整需求开发,从需求理解到最终上线,自动走完整个流程。

### [为什么要分两种?](https://stellarlink.co/articles/2025-%E6%88%91%E8%AE%A9-agent-%E8%87%AA%E5%B7%B1%E8%B7%91%E5%AE%8C%E4%BA%86#%E4%B8%BA%E4%BB%80%E4%B9%88%E8%A6%81%E5%88%86%E4%B8%A4%E7%A7%8D)

一开始想做一个”超级模式”啥都能干,试了一圈发现容易失控。比如让 AI 在分析需求的时候就能写代码,经常会跳过分析直接开始撸,导致后面逻辑混乱。

分成两种模式后,一个专注”执行”,一个专注”编排”,职责清晰多了。

## [几个踩坑经验](https://stellarlink.co/articles/2025-%E6%88%91%E8%AE%A9-agent-%E8%87%AA%E5%B7%B1%E8%B7%91%E5%AE%8C%E4%BA%86#%E5%87%A0%E4%B8%AA%E8%B8%A9%E5%9D%91%E7%BB%8F%E9%AA%8C)

### [1\. 别想着重写一套系统](https://stellarlink.co/articles/2025-%E6%88%91%E8%AE%A9-agent-%E8%87%AA%E5%B7%B1%E8%B7%91%E5%AE%8C%E4%BA%86#1-%E5%88%AB%E6%83%B3%E7%9D%80%E9%87%8D%E5%86%99%E4%B8%80%E5%A5%97%E7%B3%BB%E7%BB%9F)

一开始差点陷入”重写一套完美引擎”的陷阱。写了一个月发现越来越复杂,调试都调不动。

后来想明白了:手上已经有能用的东西了,为啥要推倒重来?在现有基础上加个轻量的编排层就够了。

### [2\. AI 得管着点,不能太自由](https://stellarlink.co/articles/2025-%E6%88%91%E8%AE%A9-agent-%E8%87%AA%E5%B7%B1%E8%B7%91%E5%AE%8C%E4%BA%86#2-ai-%E5%BE%97%E7%AE%A1%E7%9D%80%E7%82%B9%E4%B8%8D%E8%83%BD%E5%A4%AA%E8%87%AA%E7%94%B1)

最大的坑:**前期规划阶段别让 AI 写代码**。

早期版本里,AI 在分析需求的时候就能开始写代码。结果经常是分析到一半,突然忍不住开始撸代码,写完发现理解偏了,又得回头改。

后来强制规定:前期只能”看”不能”写”。分析需求时只给只读权限,等拆分完任务了再放开写代码的权限。

这个改动之后,执行准确率明显上去了。

### [3\. 并发执行得解决代码隔离](https://stellarlink.co/articles/2025-%E6%88%91%E8%AE%A9-agent-%E8%87%AA%E5%B7%B1%E8%B7%91%E5%AE%8C%E4%BA%86#3-%E5%B9%B6%E5%8F%91%E6%89%A7%E8%A1%8C%E5%BE%97%E8%A7%A3%E5%86%B3%E4%BB%A3%E7%A0%81%E9%9A%94%E7%A6%BB)

拆分出来的多个子任务要并发跑,一开始想每个任务 clone 一份代码,后来发现磁盘受不了。

改成用 git worktree 之后解决了:base repo 只 clone 一次,每个任务用独立的工作目录,互不干扰。跑完直接删工作目录,下次复用 base repo。

### [4\. 测试不能只跑后端](https://stellarlink.co/articles/2025-%E6%88%91%E8%AE%A9-agent-%E8%87%AA%E5%B7%B1%E8%B7%91%E5%AE%8C%E4%BA%86#4-%E6%B5%8B%E8%AF%95%E4%B8%8D%E8%83%BD%E5%8F%AA%E8%B7%91%E5%90%8E%E7%AB%AF)

后端测试好办,单测 + E2E 跑一遍就知道了。

前端比较麻烦,最后是让 AI 开个浏览器自己点点看,发现问题自动提 issue,然后再触发修复。这个循环跑通之后,基本就是自动闭环了。

## [基于 Claude Code 快速验证想法](https://stellarlink.co/articles/2025-%E6%88%91%E8%AE%A9-agent-%E8%87%AA%E5%B7%B1%E8%B7%91%E5%AE%8C%E4%BA%86#%E5%9F%BA%E4%BA%8E-claude-code-%E5%BF%AB%E9%80%9F%E9%AA%8C%E8%AF%81%E6%83%B3%E6%B3%95)

这个项目能跑通,最关键的是**用 Claude Code 本身来验证 MVP**。

不用从零搭环境,直接在 Claude Code 里写个 slash command 测试流程,把能力封装成 skills 随时调整。调通了之后直接复制到自己的框架里,基本不用改就能跑。

这比从零开始搭 agent 框架再一点点调试,至少快了 3-5 倍。

## [总结](https://stellarlink.co/articles/2025-%E6%88%91%E8%AE%A9-agent-%E8%87%AA%E5%B7%B1%E8%B7%91%E5%AE%8C%E4%BA%86#%E6%80%BB%E7%BB%93)

从 6 月到 12 月,花了半年把 AI 从”需要人盯着”变成”自己能跑完”。

几个关键经验:

1.  **别想着一步到位** - 先做单次任务,再做流程编排
2.  **职责分离比大而全更可控** - 执行和编排分开
3.  **用 Claude Code 快速验证** - 比从零搭框架快太多
4.  **强制阶段权限** - 别让 AI 在不该写代码的时候写代码
5.  **让 AI 选择后端** - 比人工指定更灵活

swe-agent 2.0 的 alpha 版已经在内测,有兴趣可以关注 [swe-agent.ai](https://swe-agent.ai/)。
